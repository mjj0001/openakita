# AIGC 视频创作工作室测试计划

本文面向默认组织模板 **「AIGC 视频创作工作室」**（代码常量 `aigc-video-studio`，定义见 [`src/openakita/orgs/templates.py`](../src/openakita/orgs/templates.py)）。该模板已从历史上的 **`tongyi-image` + `seedance-video` 双插件四节点** 升级为 **`happyhorse-video` 单插件七节点**：同一插件按图像 / 短视频 / 数字人 / 长视频后期拆成四个工作台叶子节点，配合制片人、编剧、美术指导三条协作角色，形成「剧本 → 分镜 JSON → 多通道生成 → 拼接成片」流水线。

> **并列说明**：仓库里仍可单独安装 `tongyi-image`、`seedance-video` 等插件或自建组织；但**内置默认「AIGC 视频创作工作室」模板不再绑定它们**。旧版手工复制的 `data/org_templates/aigc-video-studio.json` 若仍存在，`ensure_builtin_templates` **不会覆盖**，可能继续加载过时拓扑——验收前应删除该 JSON（以及已废弃的 `happyhorse-video-studio.json`）后重启后端，见 [`plugins/happyhorse-video/README.md`](../plugins/happyhorse-video/README.md) 第 7 节升级提示。

插件能力与命令速览：[README §6 CLI 示例](../plugins/happyhorse-video/README.md)；工作台 JSON 协议：`plugins/happyhorse-video/tests/test_happyhorse_workbench_protocol.py`。

## 1. 测试目标

### 1.1 用户侧怎么走

**不推荐**在用户提示里抄写「谁先派谁、必须调哪个工具」——这既脱离真实用法，也会让多数用户抓狂。验收时请以**口头就能说清的创意需求**驱动组织，看它能否**自己拆了工序、找对角色、接上资产**。

用户使用方式约定：

1. **主测试**：像对一家制作公司下 brief 那样写：**主题、给谁看、画风、时长/画幅、分几个镜头讲一下故事、要什么交付物**（脚本梗概、关键画面、小视频或一条成片均可）。不出现节点名、`org_delegate_*`、也不出现 `hh_*` 工具名 ——除非你是在做附录里的「深挖 / 工单级」抽查。
2. **验收侧**：测试人员对照「组织能力核对清单」（见 §1.2）打开组织实况、消息流或日志，判断是否**合理委派**、**资产是否串起来了**、**附件是否齐了**。
3. **可选深究**：若在自然语言链路下组织能力表现模糊，再在**单独一轮**补一条稍具体的说明（例如「希望先有一版文字分镜再出画」），仍应尽量像真用户说话，而非 API 说明书。

验证以下三条入口都能驱动同一套协作（差别只在载体）：

1. 组织编排页的「指挥台」直接下达任务。
2. 主聊天窗口通过 `@org:组织名`（必要时带 `/节点显示名`）调用组织。
3. 通过 IM 通道（飞书、钉钉、Telegram 等即时通讯入口）下达同类任务。

### 1.2 组织能力核对清单（给验收人员，不要求写进用户提示）

在自然语言驱动的前提下，实况 / 日志中**宜观察到**的组织行为如下（做不到的项记 defect，而不要埋怨用户没说清）：

1. **委派是否顺**：制片人是否先推动剧本与分镜，再把「怎么出画面、怎么动起来、要不要拼一段长的」交给统一调度角色（模板里的美术指导），而不是让用户指挥每一个节点。
2. **编剧侧结构化**：是否在合适时出现分镜拆解（可与工具 `hh_storyboard_decompose` 对应），交付中有可下发的镜头信息（例如 `segments` / 转场意图），而不只是一段散文。
3. **静帧链路**：是否在需要时出现图像类产出（`hh_image_*`），且产物可被下游引用（登记 `asset_ids` / URL）。
4. **动效链路**：若前面有关键帧，短视频侧是否用上上游资产（常见为 `hh_i2v` + **`from_asset_ids`**）；若为纯意象镜头也可能走文生短片路径。观察重点：**镜头之间是否区分开**，而非一句总 prompt 敷衍多段。
5. **收口**：多段短片是否在需要时出现拼接/后期工作台参与（例如 `hh_video_concat`，或能说清为何不拼）。
6. **Runtime 附件**：图片 / 视频是否落到 org workspace（如 `plugin_assets/`）并在交付上可见。
7. **终稿体感**：出品方拿到的不仅是聊天文字，还能看到与 brief 相称的脚本或分镜说明 + 可视化附件（图 / 小段 / 成片视任务而定）。

## 2. 术语说明

- **组织编排**：把多个 AI 节点编成上下级或协作关系，自动派工与交付验收。
- **指挥台**：组织详情页里向整个组织发任务的输入框。
- **工作台节点**：映射自插件能力集的**叶子节点**（不能再挂下属）；本模板四个工作台全部源自 **`happyhorse-video`**，仅靠各自 `external_tools` 白名单区分职责。
- **API**：后端接口；前端通过它加载模板、保存组织、启动组织。
- **IM**：即时通讯通道（飞书、钉钉等）。
- **Asset Bus（资产总线）**：插件产物在组织内的引用链路；短视频侧的 **`from_asset_ids`** 用于消费上游图像工作台登记的资产（类比「上一道工序的出库单号」）。

## 3. 模板拓扑（给验收／编辑画布用）

普通用户 briefing **不必背诵**这张图；它只是帮助测试人员在「画布 ↔ 实况消息」之间**对上人是谁、谁该接力**，判断组织是否自发协调得当。

层级（**hierarchy**，上级向下派单）简略：

```
出品方（用户 persona）
 └ 制片人（producer）
     ├ 编剧（screenwriter）
     └ 美术指导（art-director）
         ├ 图像工作台（wb-hh-image）
         ├ 短视频工作台（wb-hh-video）
         ├ 数字人工作台（wb-hh-human）
         └ 长视频后期工作台（wb-hh-long）
```

另有若干 **collaborate** 边（例如图像工作台 ↔ 短视频工作台「asset_ids 作为首帧」、各段 → 长视频后期），用于表达资产与工序衔接意图；实际 **`org_delegate_task` 仍须遵守 hierarchy 可达性**，测试中应以画布连线与模板注释为准。

七个节点的中文显示名（指挥台 / `@org` 路径常用）：

| 节点 id | 界面角色名 |
|---------|------------|
| `producer` | 制片人 |
| `screenwriter` | 编剧 |
| `art-director` | 美术指导 |
| `wb-hh-image` | 图像工作台 |
| `wb-hh-video` | 短视频工作台 |
| `wb-hh-human` | 数字人工作台 |
| `wb-hh-long` | 长视频后期工作台 |

## 4. 前置条件

### 4.1 插件：`happyhorse-video`

在「插件管理」中安装、启用并加载 **`happyhorse-video`**（快乐马工作室）。

- **DashScope**：配置可用的 API Key（环境变量常见为 `DASHSCOPE_API_KEY`，以插件设置页为准）。
- **OSS**：插件需在设置 Tab 配置可写 OSS（endpoint、bucket、AccessKey 等），用于签名 URL 与落盘链路。
- **权限**：插件 manifest 含 `assets.publish` 与 **`assets.consume`**，便于工作台之间传递 `asset_ids` / `from_asset_ids`。

验收点：

1. 插件列表中为已启用。
2. 工具列表可见核心工具：`hh_storyboard_decompose`、`hh_image_create`、`hh_i2v`、`hh_t2v`、`hh_video_concat`、`hh_status` 等（完整清单见 [`plugin.json`](../plugins/happyhorse-video/plugin.json) `provides.tools`）。

### 4.2 后端与前端

桌面版：启动 OpenAkita Desktop，确认后端正常后打开「组织编排」。

开发环境：

```bash
openakita serve
```

如需前端开发模式：

```bash
cd apps/setup-center
npm run dev:web
```

### 4.3 创建测试组织

1. 「组织编排」→「从模板创建」→ 选择 **「AIGC 视频创作工作室」**。
2. 画布核对 **恰好 7 个节点**，且四个「工作台」卡片带工作台徽章；点开任一工作台能力配置，工具列表只读。
3. 确认四个工作台**均无下属子节点**。
4. 建议固定组织名称便于 `@org` 与 IM 测试，例如：`AIGC 视频创作工作室测试`。

### 4.4 启动组织

点击「启动」，进入实况或组织消息侧栏便于观察任务流。

### 4.5 已知约束（避免误判失败）

- **DashScope 同一 API Key 异步生成并发上限约为 1**（插件侧串行）；全流程较慢属预期。
- **task_id 有约 24 h 生命周期**；隔日回归请重新生成片段再拼接。

## 5. 测试数据准备

三次入口共用同一选题，减少模型自由度带来的差异。

主题：

```text
30 秒竖屏短片：未来城市咖啡广告
```

硬性要求（写在 brief 里是合理的；不必写委派与工具名）：

```text
风格：赛博朋克、霓虹灯、雨夜、电影感
画幅：9:16 竖屏
时长：约 30 秒
镜头数：3 个（故事自己编，贴合「咖啡广告」即可）
希望在交付里能区分：先有文字上的分镜/镜头说明 → 再有画面 → 再有一段能发社交媒体的成片或小段（组织自己决定切段还是整条）
```

## 6. 路径一：组织指挥台

### 6.1 端到端（主场景：一句话 brief）

入口：打开测试组织 → 指挥台。

用户输入示例（可复制微调）：

```text
我们想拍一支大概半分钟、手机屏幕竖着看的贴片广告：未来城市里一家咖啡快闪店，主题是「霓虹雨夜喝一杯」。
画风要赛博朋克一点，霓虹灯和雨水反光要足，别太卡通。

故事你帮我想三章节奏就行：开头把人拉进街巷氛围，中间特写杯子和城市倒映，收尾有一点「城市在动、人在这刻停一下」的感觉。

交付时请给：简短的旁白或字幕口吻说明 + 能用的关键画面 + 能直接发预览的小视频或一条连贯的短片。你们内部怎么分工都行，我只要结果像一支正式提案。
```

**验收**：对照 §1.2；实况里出现多角色接力，且不依赖用户在提示里指派具体节点。**说明**：不要求用户写出「制片人 / 编剧 / 拼接」等字样；若组织最终没有视频只有文字，在缺陷单注明阶段。

可选（仅当第一轮结果太含糊时）：追加一句人话约束，而非工具名——

```text
上一版可以更「制作公司」一点：先给我一页纸能开会的分镜口述，再给图，再给视频好吗？
```

内在预期过程（仅供验收核对，不向用户口述）——制片人收 brief → 推动剧本 / 分镜 → 经由统一调度牵动各工作台出图 / 短片 /（按需）收口拼接 → 汇总交付。**允许**与理想三步顺序略有出入，关键是**可追溯、可有附件**。

### 6.2 分段验收：先要「定妆照」级别的静帧（自然语言）

用户输入示例：

```text
先别着急做动效：我想看看「未来城市咖啡广告」在赛博雨夜霓虹下，三张竖屏定妆关键帧够不够味。

三张分别是：街口拿杯广角、杯子特写倒映城市灯光、高架上看城市飞行器光轨。你们按专业习惯做主视觉就行，这轮只要高质量静帧预览，动起来以后再说。
```

**验收**：附件区出现 3 张左右竖图；实况里能看见组织为静帧发起了相应工序（§1.2 条目 3）。

### 6.3 分段验收：在手头已有静帧的前提下做动起来（尽量不粘贴 ID）

前置：上一轮 6.2 刚过，在同一组织 / 会话中继续。

用户输入示例（优先不写 asset_id）：

```text
静帧我比较满意，请在这个基础上继续做：还是那三个镜头的情绪，每张大概八九秒竖屏小视频，最后用你们习惯的方式接一下，看起来像一支能交差的 30 秒贴片。
```

若在**新会话**里测本段且历史图不可见，再用人话附带引用即可（例如附上文件），而非列数据库式 ID。

**验收**：参见 §1.2 条目 4、5；若在日志可查，可看短视频侧是否接上上游帧（常为 `hh_i2v` + **`from_asset_ids`**）；**不向普通验收用户强求**复述 `asset_ids`。

### 6.4 数字人支线（可选）

将 brief 说成品牌口播即可，不出现「数字人工作台」字眼。示例：

```text
加一个 15 秒的竖屏口播收尾：咖啡店主理人用很有温度的语气推荐「霓虹夜这款限定」，像在镜头前跟朋友说话。前面的城市氛围可以保持，但不要求露全脸也没关系，你们拿捏。
```

**验收**：实况中是否安排了口播／形象驱动类能力与后续收口（§1.2）。

### 6.5 负向：插件未启用

1. 暂时禁用 `happyhorse-video`，刷新插件状态。
2. 发送简化任务（期望系统在剧本阶段或第一台调度时报错而非崩溃）。

用户输入示例：

```text
帮做一支 15 秒竖屏的「雨夜霓虹咖啡」小广告，要真的能出几条画面和视频预览，风格和昨天聊的那种赛博夜雨一致。
```

预期：

1. 进程无崩溃。
2. 明确提示插件不可用或工具缺失。
3. 恢复启用插件后可重做 6.1。

## 7. 路径二：主聊天 `@org`

### 7.1 基础：`@org:组织名`

```text
@org:AIGC 视频创作工作室测试 我们想要一支约半分钟、9:16 的短片，未来都市咖啡快闪「霓虹雨夜喝一杯」，画风赛博朋克一点。请你们从创意到会能提案：分镜口述、关键画面、成片或小段预览，内部怎么拆开都行。
```

预期：组织实况出现与指挥台相近的阶段消息；聊天线程可见汇总与附件。**验收**仍对齐 §1.2。

### 7.2 指定制片人（仍用自然语言）

```text
@org:AIGC 视频创作工作室测试/制片人 帮出一支大概 20 秒竖屏的「清晨机器人咖啡师」贴片：轻快、晨光感、咖啡店有点科幻小道具。你给团队怎么安排我不管，我只要交付看起来像正式给客户的 DEMO。
```

预期：制片人成为入口后仍能驱动整条协作链；工作台不应要求用户点对点指派。

### 7.3 问询工作台（不应误触发）

```text
@org:AIGC 视频创作工作室测试/图像工作台 你们这类岗位一般负责到什么程度？会先出静态主视觉再给动效那边吗？我这边暂时还不想真的要图，了解下流程。
```

预期：文本答复；无新增图片附件。

```text
@org:AIGC 视频创作工作室测试/短视频工作台 如果前面美术已经出了一批气氛图，你们通常怎么接住做成镜头？用文字说明就行，这次不要生成视频。
```

预期：文本答复；无新视频附件。

## 8. 路径三：IM 通道

IM（Instant Messaging）即飞书 / 钉钉等外部客户端中的机器人链路。

### 8.1 前置

至少配置一路 IM（可参考 `docs/im-channel-setup-tutorial.md`），确认通道在线。

### 8.2 连通性探针

```text
你好，确认在线请只回复：OpenAkita IM 在线
```

### 8.3 全流程（与 6.1 等价）

```text
@org:AIGC 视频创作工作室测试 需求同我们群里刚发的「霓虹雨夜咖啡」贴片：竖屏三十秒上下，赛博城市雨夜画风。请你们内部协调出分镜说明、气氛图和视频预览，做完了在 IM 里用几句话交代交付物在哪儿即可。
```

预期：组织页与 IM 双侧可见进度；附件行为受平台发送能力限制时可降级为链接 / 路径说明（参见下文「降级判定」）。

### 8.4 指定制片人（短任务）

```text
@org:AIGC 视频创作工作室测试/制片人 想快速要一版 15 秒竖屏「发光早餐咖啡」小样，情绪温暖带一点点科幻餐具小亮点，能出预览即可，你们自行赶工顺序。
```

### 8.5 中途追问与收尾清单

中途（自然口语即可）：

```text
你先跟我说说现在进度咋样啦？稿子有了吗？图出来没？视频在制作还是已经齐了？
```

收尾（对用户友好；文件名 / 预览链接能说清就行）：

```text
最后用几条要点帮我列一下都交了啥：每一段是啥、我上哪儿能看见或点开，尽量别用大段术语。
```

内部记录时可另开一行备注：实况里可用的 `task_id`、磁盘路径或用例编号——**不要求**用户话术里带出这些字段。

## 9. 验收证据记录（建议制表）

每条用例行记录：

1. 入口类型（指挥台 / 主聊天 / IM 平台名）。
2. 组织名称。
3. 用户原始输入（保持自然语言原貌，便于评估「是否刁难式提示」）。
4. 开始 / 结束时间。
5. §1.2 清单逐项是否满足（可简写代号）；需要技术细节时再记：是否出现 `hh_storyboard_decompose`、`hh_image_*`、短片工具链、`hh_video_concat`（或跳过理由）、**`from_asset_ids`**（在静帧→动效场景）。
6. 最终附件是否齐全。
7. 失败时的报错阶段与日志关键字。

## 10. 成功判定标准

### 10.1 完整流程通过

1. **用户侧**：仅凭 §6.1 / §7.1 这类 brief **无需**写明委派关系或工具名，组织仍可跑通主要工序。
2. **组织侧**：实况里能看见多节点协作意图（而非单节点独白到底）。
3. 分镜 / 镜头信息可追溯（文稿或结构化附件皆可，不必强行要求用户可读 JSON）。
4. 图像与视频产物被 Runtime 登记（界面附件或等价引用）。
5. 多镜头体感上能区分段落，未发现「一刀切 prompt 糊弄多个镜头」的明显劣化。
6. 无 unexplained `INCOMPLETE`。

### 10.2 IM 降级仍算通过

若平台限制无法直接推送文件，但组织内已生成附件且文本中给出文件名 / 路径 / URL / task_id，可作为 IM 路径 Pass。

## 11. 常见问题排查

### 11.1 工具不可用

检查：`happyhorse-video` 是否启用；API Key / OSS 是否配置；后端是否重启加载插件。

### 11.2 图像成功但无附件

核对工具返回 JSON 是否含 URL / `asset_ids`；查看组织 workspace `plugin_assets/`；查阅后端日志下载失败与安全路径校验。

### 11.3 短视频未用上关键帧

核对 `hh_i2v` 是否传入 **`from_asset_ids`**； Asset Bus 权限是否包含 `assets.consume`；资产是否过期。

### 11.4 拼接失败

核对 `hh_video_concat` 的 **`task_ids`** 顺序与段是否已全部完成 query；转场参数是否与 `transition_to_next` 语义一致（`cut` / `crossfade` / `ai_extend`）。

### 11.5 模板仍是旧四节点

删除本地 `data/org_templates/aigc-video-studio.json`（及遗留 `happyhorse-video-studio.json`）后重启，参见插件 README 升级提示。
